# Codex 全流程提示词

## 1、需求分析

~~~
你好，我现在想做一个 xxx 系统，背景是 xxx，目标用户是 xxx，核心功能包括 xxx。

请先使用头脑风暴方式进行需求分析。
先不要写代码，也不要做技术选型。

请重点帮我澄清：

1. 系统目标
2. 用户角色
3. 核心业务流程
4. 功能模块
5. 数据对象
6. 权限边界
7. 非功能需求
8. 可能遗漏的问题
9. 暂未确定的问题

如果有关键问题，请先问我。等需求统一后，再生成本地文件：
项目需求分析文档.md
~~~

## 2、架构设计阶段

~~~
请阅读本地的 项目需求分析文档.md。
请按照 AGENTS.md 中的 Architect Mode 设计系统整体架构。

特别注意：
- 默认评估是否需要 WebSocket 实时通信能力。
- 如果 v1 需要实时通知、状态同步、后台任务进度、在线状态、实时评论或仪表盘刷新，请在架构中纳入 WebSocket。
- 如果 v1 暂不实现但后续很可能需要，请说明如何预留扩展点。
- 如果完全不需要，也要说明原因。

输出到：
- docs/系统架构/

暂时不要写业务代码。
~~~

## 3、数据库 / API / 前端设计

~~~
请按顺序依次执行三个设计步骤，每一步完成后再进入下一步。

1. 数据库设计

请阅读：
- 项目需求分析文档.md
- docs/系统架构/05-数据库选型与核心表方向.md

请按照 AGENTS.md 中的 Database Designer Reviewer Mode 设计本项目数据库。

数据库选择：
- PostgreSQL

输出：
- docs/数据库设计.md

要求：
- 只设计数据库文档，不写业务代码。
- 如发现需求文档或架构文档不足以支撑数据库设计，请先列出问题。

2. API 设计

请阅读：
- 项目需求分析文档.md
- docs/数据库设计.md
- docs/系统架构/08-API设计原则.md
- docs/系统架构/11-实时通信设计.md

请按照 AGENTS.md 中的 API Designer Mode 设计 REST API 草案。

输出：
- docs/API设计文档.md

要求：
- 只设计 API 合同，不写业务代码。
- 如发现数据库设计或需求文档不足以支撑接口，请先列出问题。
- 如果存在 WebSocket 实时事件，请说明 REST API 与实时事件的边界；不要把实时事件强行设计成普通轮询接口。

3. 前端设计

请阅读：
- DESIGN.md
- 项目需求分析文档.md
- docs/系统架构/01-系统整体架构.md
- docs/系统架构/02-前后端模块划分.md
- docs/系统架构/04-前端目录结构.md
- docs/系统架构/11-实时通信设计.md
- docs/API设计文档.md

请先进行前端视觉探索，再输出前端设计文档。

要求分两步执行：

第一步：视觉探索

请使用 imagegen 生成 2-3 个前端布局方向示意图。

要求：
- 只生成关键页面的布局方向图，不写正式前端代码。
- 遵守 DESIGN.md。
- 优先选择最能代表系统复杂度的页面，例如：
  - 首页 / 工作台 / Dashboard
  - 核心列表页
  - 核心详情页
  - 表单创建 / 编辑页
- 每个方案需要体现：
  - 页面整体布局
  - 导航结构
  - 信息密度
  - 卡片 / 表格 / 表单 / 状态区的组织方式
  - 响应式大致思路
- 不要求图片中文字完全准确，图片主要用于确认布局、视觉节奏和风格方向。
- 生成后请说明每个方案适合什么类型的系统体验，并等待我选择。

第二步：前端设计文档

在我选择视觉方向后，再生成：

- docs/前端设计文档.md

文档要求：
- 输出页面清单、路由结构、页面信息架构、组件拆分、交互状态、响应式方案。
- 标注每个页面依赖哪些 API。
- 标注 loading / empty / error / success / submitting 状态。
- 如果页面依赖实时推送，请标注 WebSocket 事件来源、断线降级方式和是否需要重新拉取 REST 数据。
- 如果 API 文档不足以支撑页面设计，请列出问题。
- 不要发明与 DESIGN.md 冲突的视觉风格。

三个步骤完成后，请汇总它们的结果，并指出前端、API、数据库之间需要对齐的字段、接口、权限和状态。
~~~

## 3.5、前后端接口对齐

~~~
请阅读：

- docs/前端设计文档.md
- docs/API设计文档.md
- docs/数据库设计.md
- docs/系统架构/02-前后端模块划分.md
- docs/系统架构/05-数据库选型与核心表方向.md
- docs/系统架构/08-API设计原则.md
- docs/系统架构/09-权限与安全设计.md
- docs/系统架构/11-实时通信设计.md

请汇总前端页面、API、数据库之间需要对齐的内容，生成：

- docs/前后端接口对齐文档.md

要求：
- 只做对齐分析和文档输出，不写业务代码。
- 如果发现冲突，先列出冲突和建议修正方案。
- 如果发现文档缺失或信息不足，请列出待确认问题。

请重点检查：

1. 每个前端页面依赖哪些 API
2. 每个 API 的请求路径
3. 每个 API 的请求方法
4. 请求参数
5. 请求体结构
6. 响应字段
7. 字段类型
8. 分页结构：page/size 或 cursor/size
9. 数字业务错误码 ResponseCode
10. 权限要求
11. loading / empty / error / success / submitting 状态来源
12. 后端数据库字段是否支撑响应字段
13. 是否有缺失 API
14. 是否有冗余 API
15. 是否存在前端字段名、API 字段名、数据库字段名不一致的问题
16. 是否存在数据库字段直接泄露到前端响应的问题
17. 是否存在前端需要的筛选/排序条件但数据库索引方向没有支撑的问题
18. 如果存在 WebSocket，检查连接路径、认证方式、事件类别、REST 数据刷新边界和断线降级方案是否一致
~~~

## 4、Planner 拆实施计划

~~~
请阅读：

- 项目需求分析文档.md
- docs/系统架构/
- docs/前端设计文档.md
- docs/API设计文档.md
- docs/数据库设计.md
- docs/前后端接口对齐文档.md

请按照 AGENTS.md 中的 Planner Mode 生成分阶段实施计划。

输出：
- 开发任务计划.md

要求：
- 只生成实施计划，不写业务代码。
- 按可运行闭环拆阶段，不要按“全部数据库 / 全部后端 / 全部前端”这种技术层拆。
- 每个阶段必须包含 backend / frontend / database / Nginx-Docker-config / verification / dependencies / parallelization / risks。
- 如发现前置文档缺失、冲突或不足以规划，请先列出问题。
~~~

## 5、/goal 实现某个阶段

~~~
请按 开发任务计划.md 中 Phase X 执行本阶段实现。

请使用 AGENTS.md 中的工作模式：
- 后端任务使用 Spring Boot Developer Mode。
- 前端任务使用 Frontend Implementer Mode。
- 数据库/SQL/mapper 风险使用 Database Designer Reviewer Mode 复核。
- 完成后使用 Before Commit Reviewer Mode 检查。

如果需要并行实现，请明确启动 Codex subagents，并保证前端、后端、数据库文档复核的写入范围互不冲突。

约束：
- 只实现 Phase X，不提前做后续阶段。
- 严格遵守 docs/API设计文档.md、docs/数据库设计.md、docs/前后端接口对齐文档.md。
- 如本阶段包含实时通信，严格遵守 docs/系统架构/11-实时通信设计.md。
- 不随意修改合同；如需修改，先报告冲突。
- 每完成一个小闭环就运行对应测试。
- 最终输出文件改动、测试结果、剩余风险。
~~~

## 7、提交前检查

~~~
请按照 AGENTS.md 中的 Before Commit Reviewer Mode 做提交前检查。

要求：
- 不修改任何文件。
- 自己运行 git status、git diff 和可用测试。
- 只 review 当前改动。
- 如有问题按 P0/P1/P2/P3 输出，并建议修复方式。
- 如无阻塞问题，明确告诉我可以进入 git add、git diff --cached、git commit。
~~~

```powershell
git status
git diff

git add .
.\scripts\format-staged-java.ps1
git add .
git diff --cached

git commit -m "feat: xxx"
git push -u origin feature/xxx
```
