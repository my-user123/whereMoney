# AI 智能记账与个人财务分析系统开发任务计划

## 1. 计划依据

本计划基于以下文档：

1. `项目需求分析文档.md`
2. `docs/系统架构/`
3. `docs/前端设计文档.md`
4. `docs/API设计文档.md`
5. `docs/数据库设计.md`
6. `docs/前后端接口对齐文档.md`

实施原则：

1. 按可运行闭环拆分阶段，不按“全部数据库 / 全部后端 / 全部前端”拆分。
2. 每个阶段都能形成可验证的用户价值。
3. 基础记账能力优先，AI、导出、实时通信作为增强能力逐步接入。
4. v1 不强制实现 WebSocket，预留扩展点，使用 REST 查询和短轮询完成状态同步。

## 2. 实施前需要锁定的问题

以下问题不阻塞总体计划，但建议在阶段 0 完成确认或以临时规则固化：

1. 账户币种与账单币种：v1 建议强制一致，前端选择账户后带出币种。
2. AI 报告内容格式：v1 建议使用 Markdown 字符串，`riskNotice` 单独字段。
3. Dashboard、图表、周期提醒实例、通知、导出记录响应结构需要补齐。
4. 详情 API 需要补齐：预算详情、储蓄目标详情、周期提醒详情、账户详情、分类详情。
5. `ResponseCode` 需要从通用码扩展为业务细分码。
6. 账单按币种筛选需要补充索引方向：`transactions(user_id, currency, occurred_at)`。
7. 导出任务 v1 是否显示百分比：建议不显示百分比，只显示任务状态。
8. 注册后默认账户和默认分类初始化策略需要确定。

## 3. 阶段 0：项目骨架与接口契约收敛

### 3.1 阶段目标

建立可启动的前后端工程骨架、本地开发环境、基础契约和数据库迁移框架，为后续纵向功能开发打地基。

### 3.2 Backend

1. 初始化 Spring Boot 3 后端工程。
2. 建立模块优先包结构：`auth`、`user`、`account`、`category`、`transaction`、`budget`、`saving`、`recurring`、`analytics`、`ai`、`notification`、`report`。
3. 建立统一响应 envelope、全局异常处理、`ResponseCode` 枚举。
4. 建立 JWT 安全基础设施和当前用户上下文。
5. 建立 MyBatis-Plus、XML mapper 目录和基础配置。
6. 补齐 API 文档中的响应结构草案，形成可实现的 DTO/VO 边界。

### 3.3 Frontend

1. 初始化 Next.js App Router 前端工程。
2. 接入 Tailwind CSS、shadcn/ui 基础组件约定。
3. 建立移动端优先的 AppShell、底部导航、桌面侧边导航。
4. 建立 API client、统一响应处理、登录态存储、请求错误处理。
5. 建立基础状态组件：loading、empty、error、success、submitting。
6. 落地 `DESIGN.md` 的基础 design tokens。

### 3.4 Database

1. 建立数据库迁移机制。
2. 创建用户、用户资料、账户、分类、账单等基础表的首批 migration。
3. 预置系统默认分类数据。
4. 增加关键索引方向，包括 `transactions(user_id, currency, occurred_at)`。

### 3.5 Nginx-Docker-config

1. 配置 Docker Compose：PostgreSQL、Redis、RabbitMQ、backend、frontend。
2. 配置 Nginx 统一入口：`/` 转发前端，`/api/` 转发后端。
3. 预留 `/ws/` 路由，但 v1 暂不启用 WebSocket。
4. 建立本地环境变量样例，禁止提交真实 `.env`。

### 3.6 Verification

1. 后端健康检查接口可访问。
2. 前端首页壳可访问。
3. Docker Compose 可启动基础依赖。
4. 数据库 migration 可重复执行。
5. 统一响应和错误码能被前端正确解析。

### 3.7 Dependencies

1. 无业务阶段依赖。
2. 依赖 AGENTS.md 默认栈。
3. 依赖接口对齐文档中待定项收敛。

### 3.8 Parallelization

1. 后端工程骨架和前端工程骨架可并行。
2. 数据库 migration 可与后端实体/mapper 并行，但字段名需每日对齐。
3. Nginx/Docker 配置可独立并行。

### 3.9 Risks

1. 如果 API 响应结构不收敛，后续前端类型会频繁返工。
2. 如果默认分类初始化策略不明确，注册后的首屏体验会断裂。
3. 如果权限上下文晚于业务开发接入，后续资源归属校验会返工。

## 4. 阶段 1：用户登录与基础账本闭环

### 4.1 阶段目标

用户可以注册、登录，创建账户和分类，完成手动收入/支出记账，并在账单列表和基础工作台看到结果。

### 4.2 Backend

2. 注册后初始化默认分类。
3. 实现账户 CRUD 和归档。
4. 实现分类 CRUD 和停用。
5. 实现手动账单创建、详情、列表、编辑、删除。
6. 实现账单创建/编辑/删除与账户余额的事务一致性。
7. 实现基础 Dashboard 摘要：按币种返回收入、支出、结余、储蓄率、近期账单。
8. 所有资源接口接入当前用户归属校验。

### 4.3 Frontend

1. 实现登录页和注册页。
2. 实现移动端 AppShell、底部导航和桌面侧边导航。
3. 实现账户管理页面。
4. 实现分类管理页面。
5. 实现账单列表、筛选、创建、编辑、删除。
6. 实现基础首页工作台：财务摘要、近期账单、快捷记账入口。
7. 覆盖 loading、empty、error、success、submitting 状态。

### 4.4 Database

1. 完成 `users`、`user_profiles`、`accounts`、`categories`、`transactions` 表。
2. 建立用户内资源归属索引。
3. 建立账单高频查询索引。
4. 明确软删除查询策略。

### 4.5 Nginx-Docker-config

1. 打通前端到 `/api/v1` 的代理。
2. 配置本地 CORS 或统一入口代理，避免跨域阻塞。
3. 配置前端环境变量指向 API 基础路径。

### 4.6 Verification

1. 注册新用户后能登录。
2. 新用户能看到默认分类。
3. 用户能创建账户。
4. 用户能创建收入和支出账单。
5. 账单创建后账户余额正确变化。
6. 编辑和删除账单后余额正确回滚或重算。
7. A 用户不能访问 B 用户资源。
8. 前端移动端和桌面端基础布局可用。

### 4.7 Dependencies

1. 阶段 0 完成。
2. 默认分类清单确定。
3. 账户币种与账单币种关系确定为 v1 强制一致。

### 4.8 Parallelization

1. 认证/用户模块与账户/分类模块可并行。
2. 前端可先基于 mock 类型搭建页面，再接真实 API。
3. 账单事务逻辑应由一个后端开发独立负责，避免余额规则分散。

### 4.9 Risks

1. 账单编辑和删除容易造成账户余额不一致，需要重点测试。
2. 多币种展示口径不清会误导用户。
3. 前端如果长期缓存财务明细，可能出现删除后状态不一致。

## 5. 阶段 2：预算、储蓄目标、周期提醒与通知闭环

### 5.1 阶段目标

用户可以创建预算和储蓄目标，设置周期账单提醒，到期后确认入账，并在通知中心看到提醒。

### 5.2 Backend

1. 实现预算列表、创建、详情、更新、删除、预算执行详情。
2. 实现储蓄目标列表、创建、详情、更新、删除和进度计算。
3. 实现周期账单提醒规则 CRUD。
4. 实现周期账单实例生成、查询、确认入账、跳过。
5. 实现通知创建、列表、未读数、已读、全部已读、删除。
6. 实现预算接近阈值和超额提醒生成。
7. 实现周期提醒扫描任务。

### 5.3 Frontend

1. 实现预算页面和预算详情/编辑。
2. 实现储蓄目标页面和目标详情/编辑。
3. 实现周期提醒页面、到期实例列表、确认入账表单、跳过操作。
4. 实现通知中心和首页未读数量。
5. 首页工作台加入预算进度、储蓄目标、周期账单提醒入口。
6. 完整处理 empty、processing、submitting、error 状态。

### 5.4 Database

1. 完成 `budgets`、`saving_goals`、`recurring_bill_reminders`、`recurring_bill_instances`、`notifications` 表。
2. 建立周期提醒扫描索引。
3. 建立通知未读和类型筛选索引。
4. 确保周期实例同一规则同一日期唯一。

### 5.5 Nginx-Docker-config

1. 确认后端定时任务在本地和容器中可运行。
2. Redis 可用于通知未读数缓存或任务辅助状态。
3. RabbitMQ 可暂不作为强依赖；若启用通知异步生成，需配置队列。

### 5.6 Verification

1. 创建预算后，账单支出能正确反映预算进度。
2. 达到预算阈值后能生成通知。
3. 创建储蓄目标后，进度能按结余计算。
4. 创建周期提醒后，到期能生成实例和通知。
5. 确认周期账单后生成正式账单，且不会重复入账。
6. 跳过周期实例后不会生成账单。
7. 通知已读、全部已读、删除逻辑正确。

### 5.7 Dependencies

1. 阶段 1 的账单、账户、分类闭环完成。
2. 预算阈值默认值和范围确定。
3. 周期规则 v1 支持范围确定。

### 5.8 Parallelization

1. 预算与储蓄目标可并行。
2. 周期提醒和通知可并行，但确认入账需依赖账单服务。
3. 前端预算/储蓄页面可与后端接口并行，周期提醒确认页需等确认入账契约稳定。

### 5.9 Risks

1. 周期提醒若无实例去重，容易重复提醒或重复入账。
2. 预算统计若未按币种隔离，会产生错误使用率。
3. 通知中心如果只依赖实时推送，离线用户会漏消息；v1 必须落库。

## 6. 阶段 3：统计可视化与首页工作台增强闭环

### 6.1 阶段目标

用户可以通过首页和统计页理解消费趋势、分类结构、消费热力图、预算执行和储蓄目标状态。

### 6.2 Backend

1. 补齐 Dashboard 完整响应结构。
2. 实现收支趋势 API。
3. 实现分类占比 API。
4. 实现消费热力图 API。
5. 实现分类排行 API。
6. 统一多币种统计口径：按币种分组或指定币种过滤。
7. 根据数据量决定是否引入统计快照。

### 6.3 Frontend

1. 首页工作台升级为完整视觉方案 1。
2. 实现趋势折线图、分类占比图、热力图预览和入口。
3. 实现统计分析页：周期选择、币种选择、趋势、饼图、热力图、排行。
4. 实现图表 loading、empty、error 状态。
5. 图表与筛选条件联动。

### 6.4 Database

1. 评估并补充统计查询索引。
2. 增加 `transactions(user_id, currency, occurred_at)`。
3. 如性能不足，落地 `finance_stat_snapshots`。
4. 明确统计缓存或快照失效规则。

### 6.5 Nginx-Docker-config

1. 根据图表接口响应体大小调整代理超时和压缩策略。
2. 若引入 Redis 缓存统计结果，配置缓存连接和过期策略。

### 6.6 Verification

1. 新增账单后 Dashboard 摘要刷新正确。
2. 删除账单后图表统计同步变化。
3. 不同币种数据不错误合并。
4. 无账单周期展示 empty 状态。
5. 图表在移动端不溢出、不遮挡。
6. 账单列表与统计结果口径一致。

### 6.7 Dependencies

1. 阶段 1、阶段 2 的账单、预算、储蓄数据稳定。
2. 图表 API 响应结构完成。
3. 多币种统计口径确认。

### 6.8 Parallelization

1. 后端趋势/分类/热力图 API 可分工并行。
2. 前端图表组件可与 API 并行开发，使用 mock 数据先搭建。
3. 首页工作台增强与统计页可并行，但共享图表组件需要统一接口。

### 6.9 Risks

1. 统计口径与账单列表口径不一致会造成用户不信任。
2. 图表在移动端信息密度过高，可能影响可读性。
3. 数据量增长后聚合查询可能变慢，需要预留缓存或快照。

## 7. 阶段 4：AI 自然语言记账与财务助手闭环

### 7.1 阶段目标

用户可以通过自然语言完成记账确认，也可以手动触发 AI 月度总结、消费分析、预算建议、趋势预测和问答。

### 7.2 Backend

1. 实现 AI 服务适配层。
2. 实现自然语言记账解析 API。
3. 实现 AI 解析记录保存。
4. 实现解析结果确认入账，复用账单创建事务。
5. 实现 AI 报告任务创建、查询、列表、删除。
6. 实现 AI 问答记录创建、列表、删除。
7. AI 输出统一追加风险提示。
8. 账单变更后将受影响 AI 报告标记为 `STALE`。

### 7.3 Frontend

1. 实现 AI 助手页面。
2. 实现自然语言输入和解析候选确认界面。
3. 实现 AI 问答界面和历史列表。
4. 实现 AI 报告列表、详情、失效状态、重新生成入口。
5. 首页显示 AI 洞察摘要或生成入口。
6. 所有 AI 交互覆盖 submitting、processing、stale、error 状态。

### 7.4 Database

1. 完成 `ai_parse_records`、`ai_reports`、`ai_chat_records` 表。
2. 确认 `ai_reports.content` 使用 Markdown。
3. 确认 `currency` 是否存入 `input_snapshot`。
4. 建立 AI 报告按用户、类型、周期查询索引。

### 7.5 Nginx-Docker-config

1. 配置 AI 服务调用所需环境变量占位，但不提交真实密钥。
2. 根据 AI 响应耗时调整后端超时。
3. 若 AI 报告异步化，引入 RabbitMQ 任务队列。

### 7.6 Verification

1. 输入自然语言后返回候选账单。
2. 用户修改候选结果后确认入账成功。
3. AI 解析失败不影响手动记账。
4. 生成月度总结后能查看报告。
5. 删除账单后相关报告标记失效。
6. AI 问答只能读取当前用户数据。
7. AI 输出包含风险提示。

### 7.7 Dependencies

1. 阶段 1 的账单闭环稳定。
2. 阶段 3 的统计聚合可用于 AI 上下文。
3. AI provider、密钥配置和调用策略确定。

### 7.8 Parallelization

1. AI 自然语言记账与 AI 报告可并行。
2. 前端 AI 页面可先接 mock 响应开发交互。
3. 报告异步任务和问答同步接口可分开实现。

### 7.9 Risks

1. AI 解析结果不稳定，必须用户确认后入账。
2. AI 报告内容如果无结构约定，前端展示会困难。
3. AI 调用失败、超时或限流不能阻塞基础记账。
4. AI 上下文不能泄露其他用户财务数据。

## 8. 阶段 5：月报导出与异步任务闭环

### 8.1 阶段目标

用户可以选择周期和格式导出 Excel/PDF 月报，查看导出历史和任务状态，并下载生成结果。

### 8.2 Backend

1. 实现导出任务创建。
2. 实现导出记录列表、详情、删除。
3. 实现 Excel 月报生成。
4. 实现 PDF 月报生成。
5. 实现导出文件鉴权下载。
6. 实现导出失败状态和错误信息。
7. 导出完成后生成站内通知。

### 8.3 Frontend

1. 实现月报导出页面。
2. 实现导出周期、币种、格式、是否包含 AI 总结选项。
3. 实现导出历史列表。
4. 实现任务状态展示：PENDING、PROCESSING、SUCCESS、FAILED。
5. 实现下载按钮和失败重试入口。
6. 首页快捷入口接入导出任务创建。

### 8.4 Database

1. 完成 `export_records` 表。
2. 确认不向前端暴露 `file_path`。
3. 增加文件过期时间和清理策略。
4. 根据需要保存导出时的数据范围和摘要。

### 8.5 Nginx-Docker-config

1. 配置本地文件存储目录或对象存储适配。
2. 配置下载接口代理和超时。

### 8.6 Verification

1. 创建导出任务后能进入 PROCESSING。
2. 成功生成 Excel 文件并可下载。
3. 成功生成 PDF 文件并可下载。
4. 未登录或非归属用户不能下载文件。
5. 导出失败时前端显示失败原因。

### 8.7 Dependencies

1. 阶段 3 统计数据稳定。
2. 阶段 4 AI 报告可选接入。
3. RabbitMQ 或后端异步任务机制可用。

### 8.8 Parallelization

1. Excel 和 PDF 生成可并行。
2. 前端任务列表和后端任务状态接口可并行。
3. 文件下载鉴权应由后端安全负责人统一实现。

### 8.9 Risks

1. 导出文件包含敏感财务数据，下载鉴权必须严格。
2. PDF 图表截图或服务端图表生成可能增加实现复杂度。
3. 大报表生成耗时较长，需要避免阻塞请求线程。

3. 完善短轮询和手动刷新机制。
4. 优化表单校验、错误提示和确认弹窗。
5. 检查无数据、新用户、AI 失败、导出失败等边界体验。

### 9.4 Database

1. 检查慢查询和索引命中。
2. 检查软删除数据查询是否默认过滤。
3. 检查金额精度和多币种字段。
4. 制定导出文件记录清理策略。

### 9.5 Nginx-Docker-config

1. 完善生产/本地配置分离。
2. 检查 `/api/`、`/`、预留 `/ws/` 代理配置。
3. 配置基础压缩、超时、请求体限制。
4. 完善 Docker Compose 启动顺序和健康检查。

### 9.6 Verification

1. 全流程冒烟：注册、登录、建账户、建账单、看首页、设预算、确认周期账单、AI 分析、导出月报。
2. 权限测试：跨用户资源访问必须失败。
3. 多币种测试：不同币种不错误合并。
4. 移动端布局测试：关键页面无横向溢出。
5. 异步任务测试：AI 和导出失败/成功状态可见。
6. 安全测试：导出文件不能绕过鉴权下载。

### 9.7 Dependencies

1. 阶段 1 到阶段 5 核心闭环完成。
2. 有足够测试数据覆盖统计和 AI 分析。
3. 是否启用 WebSocket 取决于实际等待时间和体验反馈。

### 9.8 Parallelization

1. 前端视觉 QA、后端性能检查、数据库索引检查可并行。
2. WebSocket 预研可独立进行，不阻塞 v1 发布。
3. 测试用例整理可与缺陷修复并行。

### 9.9 Risks

1. 如果在 v1 末期临时启用 WebSocket，可能引入连接管理和消息兜底复杂度。
2. 图表、AI、导出三类耗时能力叠加后可能暴露性能问题。
3. 如果 ResponseCode 不统一，前端错误处理会碎片化。

## 10. 阶段间依赖总览

```text
阶段 0：工程骨架与契约收敛
  -> 阶段 1：用户登录与基础账本闭环
    -> 阶段 2：预算、储蓄目标、周期提醒与通知闭环
      -> 阶段 3：统计可视化与首页工作台增强闭环
        -> 阶段 4：AI 自然语言记账与财务助手闭环
          -> 阶段 5：月报导出与异步任务闭环
            -> 阶段 6：体验收尾、质量加固与 WebSocket 扩展评估
```

说明：

1. 阶段 2 可以在阶段 1 后半段并行启动部分数据库和页面骨架。
2. 阶段 3 的前端图表组件可以提前 mock 开发，但真实数据依赖阶段 1 和阶段 2。
3. 阶段 4 的 AI UI 可以提前 mock，但确认入账必须依赖阶段 1 账单事务。
4. 阶段 5 的导出依赖阶段 3 的统计口径和阶段 4 的 AI 报告可选接入。
