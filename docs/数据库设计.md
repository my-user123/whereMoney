# 数据库设计文档

## 1. 设计范围

本文档基于《项目需求分析文档.md》和《docs/系统架构/05-数据库选型与核心表方向.md》设计第一阶段数据库。第一阶段聚焦单人个人记账，不设计信用卡、账户间转账、退款、报销、借入借出、家庭共享账本、账单导入、OCR、资产负债管理和后台管理员系统。

数据库选择 PostgreSQL。

## 2. 设计假设与 Phase 0 锁定规则

Phase 0 已锁定以下建模规则：

1. 多币种统计第一阶段按原币种分别统计，暂不强制做汇率换算。
2. 若后续做汇率换算，需要新增汇率表或汇率快照，避免历史统计随当前汇率变化。
3. 储蓄目标进度第一阶段按用户周期内结余计算，不额外设计手动存入流水。
4. 预算提醒阈值第一阶段支持默认阈值，并在预算表预留用户自定义阈值。
5. 提醒渠道第一阶段以站内通知为主，外部推送、短信、邮件只预留类型字段。
6. AI 报告保留历史记录，但当账单变更影响报告数据范围时标记失效。
7. 导出月报文件不直接公开访问，需要通过后端鉴权下载。
8. 账户币种与账单币种 v1 强制一致，账单仍保存 `currency` 以支撑多币种筛选和后续扩展。
9. 注册后自动初始化默认分类；实现上复制为用户自己的分类记录，`is_system = true` 表示来源于系统预置，不使用全局共享分类参与写入。
10. 注册后不自动创建默认账户；用户需要手动创建账户后才能正式记账。

## 3. 通用建模规范

### 3.1 主键

业务表统一使用雪花 ID、UUID 或数据库生成 ID 均可。文档以 `id` 表示主键，不限定具体生成策略。

### 3.2 用户归属

除系统级默认配置表外，所有用户业务表必须包含 `user_id`，用于数据隔离和权限校验。

### 3.3 金额与币种

1. 金额使用 PostgreSQL 精确数值类型，例如 `numeric(18,2)` 或按业务需要扩大精度。
2. 币种使用 ISO 4217 货币代码，例如 `CNY`、`USD`、`JPY`。
3. 第一阶段保留原始币种金额，不把所有金额强制换算为默认币种。

### 3.4 时间字段

所有核心业务表建议包含：

1. `created_at`
2. `updated_at`
3. `deleted_at`，需要软删除的业务表使用

业务发生时间单独建模，例如账单的 `occurred_at`、提醒的 `next_remind_at`。

### 3.5 删除策略

财务核心数据建议默认软删除，便于审计和误删恢复。用户界面表现为删除，统计查询默认排除已删除数据。

### 3.6 状态字段

状态字段统一使用明确枚举值，不使用模糊布尔组合表达复杂状态。

## 5. 表设计

### 5.1 用户表 users

用途：保存登录用户的基础账号信息。

核心字段方向：

| 字段 | 说明 |
| --- | --- |
| id | 用户 ID |
| username | 用户名或登录名 |
| email | 邮箱，可选 |
| phone | 手机号，可选 |
| password_hash | 密码哈希 |
| status | 用户状态 |
| last_login_at | 最近登录时间 |
| created_at | 创建时间 |
| updated_at | 更新时间 |
| deleted_at | 删除时间 |

约束与索引：

1. `username` 唯一。
2. `email`、`phone` 如启用登录能力，应分别唯一。
3. `status` 用于账号禁用和注销。

### 5.2 用户资料表 user_profiles

用途：保存用户展示资料和偏好。

核心字段方向：

| 字段 | 说明 |
| --- | --- |

| user_id | 用户 ID |
| nickname | 昵称 |
| avatar_url | 头像地址 |
| default_currency | 默认币种 |

约束与索引：

1. `user_id` 唯一。
2. `default_currency` 不代表所有统计都必须换算，仅作为默认展示和新建表单默认值。

用途：保存用户资金账户。第一阶段账户仅用于收入/支出的资金来源，不支持账户间转账和信用卡逻辑。

核心字段方向：

1. 索引：`user_id`、`user_id + status`。
2. 同一用户下账户名称建议唯一或软唯一。
3. 当前余额由账单写入链路维护，统计时以正式账单为准进行校验。

核心字段方向：

| 字段 | 说明 |

1. 索引：`user_id + type + status`。
2. 同一用户同一类型下分类名称建议唯一或软唯一。

### 5.5 账单表 transactions

用途：保存正式收入和支出记录，是统计、预算、储蓄目标和 AI 分析的事实来源。

核心字段方向：

| 字段 | 说明 |
| --- | --- |
| id | 账单 ID |
| user_id | 用户 ID |
| account_id | 账户 ID |
| category_id | 分类 ID |
| type | 收入或支出 |
| amount | 金额 |
| currency | 币种 |
| occurred_at | 发生时间 |
| note | 备注 |
| source | 账单来源 |
| source_ref_id | 来源记录 ID，例如 AI 解析记录或周期提醒记录 |
| created_at | 创建时间 |
| updated_at | 更新时间 |
| deleted_at | 删除时间 |

约束与索引：

1. 金额必须大于 0。
2. 账户、分类必须属于当前用户或系统默认可用范围。
3. 索引：`user_id + occurred_at`。
4. 索引：`user_id + category_id + occurred_at`。
5. 索引：`user_id + account_id + occurred_at`。
6. 索引：`user_id + type + occurred_at`。
7. 索引：`user_id + currency + occurred_at`。

一致性规则：

1. 新增收入增加账户余额，新增支出减少账户余额。
2. 编辑账单时需要回滚旧影响再应用新影响。
3. 删除账单时需要回滚账户余额，并使相关统计缓存和 AI 报告失效。

## 6. 关系设计

1. users 1:1 user_profiles。
2. users 1:N accounts。
3. users 1:N categories。
4. users 1:N transactions。
5. accounts 1:N transactions。
6. categories 1:N transactions。
7. users 1:N budgets。
8. categories 1:N budgets，分类预算可选。
9. users 1:N saving_goals。
10. recurring_bill_reminders 1:N recurring_bill_instances。
11. recurring_bill_instances 0:1 transactions，确认入账后关联正式账单。
12. ai_parse_records 0:1 transactions，确认入账后关联正式账单。
13. users 1:N notifications。
14. users 1:N ai_reports。
15. users 1:N ai_chat_records。
16. users 1:N export_records。

## 7. 索引设计方向

### 7.1 高频查询索引

1. 账单按用户和时间查询：`transactions(user_id, occurred_at)`。
2. 账单按用户、分类、时间统计：`transactions(user_id, category_id, occurred_at)`。
3. 账单按用户、账户、时间统计：`transactions(user_id, account_id, occurred_at)`。
4. 通知按用户和未读状态查询：`notifications(user_id, read_at, created_at)`。
5. 预算按用户和状态查询：`budgets(user_id, status)`。
6. 周期提醒按到期时间扫描：`recurring_bill_reminders(status, next_remind_at)`。
7. AI 报告按用户、类型、周期查询：`ai_reports(user_id, report_type, period_start, period_end)`。
8. 导出记录按用户和状态查询：`export_records(user_id, status, created_at)`。
9. 账单按用户、币种、时间查询：`transactions(user_id, currency, occurred_at)`。

### 7.2 软删除索引注意

软删除表的常用查询需要默认过滤 `deleted_at is null`。PostgreSQL 可考虑部分索引优化未删除数据查询。

## 8. 约束设计方向

1. 金额字段必须大于 0。
2. 预算结束日期必须晚于开始日期。
3. 储蓄目标目标日期必须晚于开始日期。
4. 账单类型与分类类型应一致。
5. 账户币种与账单币种 v1 强制一致；如果后续允许不同币种账户入账，需要明确换算规则。
6. 周期账单实例同一规则同一日期不能重复。
7. 用户自定义分类和账户名称建议做用户内软唯一。

## 9. 数据一致性与事务

### 9.1 账单创建

账单创建与账户余额更新应在同一事务内完成。事务完成后再触发统计缓存失效、预算提醒判断和通知事件。

### 9.2 账单编辑

账单编辑需要比较旧账单和新账单，回滚旧账户余额影响，再应用新账单影响。若账户、金额、类型或币种变更，需要特别校验。

### 9.3 账单删除

账单删除需要回滚账户余额，软删除账单，使统计缓存失效，并将受影响周期内的 AI 报告标记为失效或删除。

### 9.4 周期提醒确认入账

周期提醒实例确认时，必须在事务中创建正式账单，并更新实例状态为 `CONFIRMED`，防止重复入账。

### 9.5 AI 解析确认入账

AI 解析记录与正式账单分离。AI 解析成功只生成候选结果，用户确认后才进入账单创建事务。

## 10. 多币种设计

第一阶段保守策略：

1. 账单、账户、预算、储蓄目标均保存币种。
2. 统计默认按币种分别展示。
3. 不强制做跨币种总和。
4. 用户默认币种只作为新建默认值和展示偏好。

后续如需要换算：

1. 新增汇率表或汇率快照。
2. 账单可保存发生时汇率快照。
3. 报表和 AI 分析明确使用何种汇率口径。

## 11. 数据风险与迁移风险

1. 如果第一阶段忽略多币种口径，后续统计和 AI 分析容易产生误导。
2. 如果账户余额只依赖增量更新，历史账单修正可能导致余额不一致，需要提供重算能力。
3. 如果 AI 报告不记录数据范围，用户删除账单后报告可能继续展示过期结论。
4. 如果导出文件无过期和鉴权设计，可能泄露财务数据。
5. 如果周期提醒不设计实例表，容易产生重复提醒或重复入账。
6. 如果软删除数据过多，后续需要归档或清理策略。
