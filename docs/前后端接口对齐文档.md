# 前后端接口对齐文档

## 1. 文档范围

本文档基于以下资料，对前端页面、REST API、数据库设计、权限安全和实时通信边界进行接口对齐分析：

1. `docs/前端设计文档.md`
2. `docs/API设计文档.md`
3. `docs/数据库设计.md`
4. `docs/系统架构/02-前后端模块划分.md`
5. `docs/系统架构/05-数据库选型与核心表方向.md`
6. `docs/系统架构/08-API设计原则.md`
7. `docs/系统架构/09-权限与安全设计.md`
8. `docs/系统架构/11-实时通信设计.md`

本阶段只做对齐分析，不写业务代码。

## 2. 总体结论

当前前端页面、API 草案和数据库设计整体方向一致，能够支撑第一阶段的单人个人智能记账系统。

已对齐内容：

1. 核心资源已覆盖：用户、账户、分类、账单、预算、储蓄目标、周期账单提醒、通知、AI 解析、AI 报告、AI 问答、导出记录、统计分析。
2. API 基础路径统一为 `/api/v1`。
3. 分页统一使用 `page` + `size`，未引入 cursor 分页。
4. 权限边界统一：前端不传 `userId`，后端从 JWT 认证上下文获取当前用户。
5. v1 不强制 WebSocket，使用 REST 查询和短轮询；后续 WebSocket 只作为通知和刷新信号。
6. 数据库字段能支撑绝大多数前端展示字段，且没有要求直接暴露数据库实体。

主要风险：

1. 部分后续阶段接口仍需在实现前补充更细的校验错误示例。
2. `ResponseCode` 已锁定业务码方向，但预算、周期提醒、AI、导出等模块要等对应阶段实现时接入代码。
3. PDF 月报图表生成方式和 AI provider 配置仍未进入实现层。
4. Compose 仍保留模板 MySQL 服务，本项目当前不使用，需要在后续环境清理时决定是否移除。

## 3. 冲突与建议修正方案

### 3.1 登录/注册状态命名冲突

冲突：

前端文档同时使用 `loading` 和 `submitting` 表示登录/注册提交中。API 侧没有区分。

建议：

1. 页面初次拉取数据使用 `loading`。
2. 表单提交统一使用 `submitting`。
3. 登录/注册页不需要 `loading`，除非后续有进入页面自动校验登录态。

### 3.2 账户币种与账单币种关系未定

冲突：

数据库建议账户币种与账单币种默认一致，但 API 创建账单允许传入 `currency`，未明确是否必须与账户币种一致。

建议：

1. v1 强制账单币种等于账户币种。
2. 前端选择账户后自动带出币种，不允许单独修改账单币种。
3. 如后续允许跨币种账户入账，需要新增汇率或换算规则。

### 3.3 AI 报告内容结构未定

冲突：

API 文档中 `ai_reports.content` 为空字符串示例；前端需要展示 AI 总结、预算建议、趋势预测和风险提示，但不知道 `content` 是 Markdown、纯文本还是结构化 JSON。

建议：

1. v1 使用 Markdown 字符串，便于快速渲染段落、列表和强调。
2. `riskNotice` 单独字段保留，不混入正文。
3. 后续如需要报告卡片化展示，再升级为结构化 JSON。

### 3.4 导出任务是否有进度字段未定

冲突：

前端设计中有 `processing` 状态和任务状态行；API 只有 `status`，没有 `progress`。

建议：

1. v1 仅展示 `PENDING`、`PROCESSING`、`SUCCESS`、`FAILED` 状态，不展示百分比。
2. 如后续需要进度条，在 `export_records` 和 AI 报告任务响应中增加 `progress`。

### 3.5 周期提醒实例 API 响应不足

冲突：

前端需要展示到期提醒实例的名称、金额、币种、账户、分类、预计发生日期，但 API 只定义了列表路径，未定义响应项。

建议：

补充 `GET /api/v1/recurring-bill-instances` 响应项，包含实例字段和提醒规则摘要。

### 3.6 首页仪表盘响应不足

冲突：

前端首页依赖预算亮点、储蓄目标亮点、近期账单和 AI 洞察，但 API 示例中相关字段为空数组或空字符串。

建议：

补充 `budgetHighlights`、`savingGoalHighlights`、`recentTransactions`、`aiInsightSummary` 的明确结构。

## 4. 待确认问题

Phase 0 已锁定以下问题，后续实现以本节结论为准：

1. 账户币种与账单币种：v1 强制一致，前端选择账户后带出币种，后端最终校验。
2. AI 报告内容：v1 使用 Markdown 字符串，`riskNotice` 单独返回。
3. 导出任务进度：v1 不显示百分比，仅展示任务状态。
4. 默认初始化：注册后自动初始化默认分类；不自动创建默认账户。
5. 多币种统计：不传 `currency` 时按币种分组返回；不做跨币种汇总换算。
6. 软删除后的资源再次访问：按资源不存在处理，业务响应码使用对应资源不存在码或 404。
7. 账单默认排序：`occurredAt desc, createdAt desc`。

仍需后续阶段继续确认：

1. 预算提醒阈值 `warningThreshold` 的默认值和可选范围。
2. AI 报告是否允许用户重新生成并保留历史版本。
3. 通知列表是否需要按 `relatedType` 筛选。
4. 导出文件过期后前端显示方式。
5. AI 问答是否需要异步任务化；当前 API 按同步返回设计。

## 8. ResponseCode 对齐

当前 API 文档只定义通用 code：

| code | 含义 |

Phase 0 已锁定业务细分码方向，后端 `ResponseCode` 枚举按实现阶段逐步补齐。当前 Phase 1 已覆盖用户、账户、分类、账单和权限相关业务码；预算、周期提醒、AI、导出业务码在对应阶段实现时补齐。
