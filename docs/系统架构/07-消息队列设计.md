# 07-消息队列设计

## 1. 是否需要消息队列

第一阶段建议引入消息队列作为异步解耦能力，但核心记账链路不依赖消息队列完成事务。

适合异步处理的场景：

1. AI 月度总结生成。
2. AI 消费行为分析。
3. AI 预算建议和趋势预测。
4. Excel/PDF 月报导出。
5. 周期账单提醒扫描后的通知生成。
6. 预算提醒通知生成。
7. 统计缓存刷新或统计快照生成。

## 2. 消息队列边界

1. 账单新增、编辑、删除的核心事务不通过消息队列保证。
2. 消息队列用于后置处理，例如通知、报告、导出和缓存刷新。
3. 异步任务必须具备失败状态和重试策略。
4. 消息消费要考虑幂等，避免重复通知、重复导出或重复生成报告。

## 3. 事件方向

架构上可抽象以下事件方向：

1. 账单变更事件。
2. 预算接近阈值事件。
3. 周期账单到期事件。
4. AI 任务创建事件。
5. 导出任务创建事件。
6. 通知创建事件。

以上是事件方向，不在架构阶段展开具体消息字段。

## 4. 与通知系统的关系

消息队列消费后生成站内通知。后续如果启用 WebSocket，通知生成后可额外推送实时事件，但站内通知仍作为离线兜底。

## 5. 与 AI 的关系

AI 任务可能响应时间不稳定，适合通过异步任务处理。用户触发后获得任务状态，系统在任务完成后保存报告并通知用户。
