# 08-API设计原则

## 1. 基本原则

REST API 作为第一阶段前后端主要交互方式。API 设计应围绕资源和业务动作展开，避免直接暴露数据库结构。

基础路径统一使用 `/api/v1`。

## 2. 响应规范

使用统一响应信封：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

业务错误使用数字业务码，并与后端统一响应码保持一致。

## 3. 认证规范

需要登录的接口使用 JWT 认证，前端通过：

```text
Authorization: Bearer <token>
```

传递登录凭证。

## 4. 资源命名原则

1. 使用复数名词表达资源。
2. 不暴露数据库实体名称作为接口契约。
3. AI、导出、提醒等动作型能力也应围绕资源状态表达。
4. WebSocket 事件与 REST API 分离，不混用语义。

## 5. 分页原则

默认使用 `page` 和 `size` 进行分页，返回结果包含总数和列表。账单列表、通知列表、AI 问答历史、导出记录等适合分页。

游标分页仅在后续高频追加数据、消息流或无限滚动场景中考虑。

## 6. 数据安全原则

1. 所有需要登录的接口必须从认证上下文获得用户身份。
2. 用户不得通过请求参数访问其他用户数据。
3. 后端必须校验资源归属。
4. 导出文件下载必须校验用户身份和文件归属。
5. AI 问答和分析接口只能访问当前用户授权范围内的数据。

## 7. AI API 原则

1. 自然语言记账接口只返回解析候选，不能直接生成正式账单。
2. 生成正式账单必须由用户确认触发。
3. AI 分析、预算建议和趋势预测应保留任务状态或报告状态。
4. AI 输出必须包含风险提示。

## 8. 异步任务 API 原则

导出和较重的 AI 分析可以采用任务模型。用户发起任务后查询任务状态，完成后获取结果或下载文件。

## 9. WebSocket API 原则

v1 可不实现 WebSocket。后续如实现，路径通过 `/ws/` 统一进入。WebSocket 只推送通知、任务状态和轻量刷新信号，不承载账单创建、删除、预算修改等核心写操作。
