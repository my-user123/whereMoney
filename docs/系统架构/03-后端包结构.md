# 03-后端包结构

## 1. 包结构原则

后端采用模块优先的包结构。每个业务模块内部按 controller、service、dto、vo、mapper、entity、convert、event 等职责划分，避免所有控制器、服务和实体堆叠在同一层级。

基础能力放在 common 和 infrastructure 下，业务能力放在 modules 下，定时任务和脚本放在 job、script 下。

## 3. 模块说明

### 3.1 common

存放跨模块基础能力，包括统一响应、异常处理、枚举、常量、安全上下文、通用校验和工具类。

### 3.2 infrastructure

存放技术基础设施适配，包括数据库增强、缓存封装、消息队列、AI 服务适配、文件存储、导出组件和预留 WebSocket 支撑。

### 3.3 modules

存放业务模块。模块之间通过 service 或事件协作，避免直接跨模块操作 mapper。

### 3.4 job

存放定时任务，例如周期账单提醒扫描、月度总结提醒触发、导出任务清理等。

### 3.5 script

存放一次性脚本或运维辅助脚本，不承载核心业务逻辑。

## 4. 事务边界

1. 账单新增、编辑、删除与账户余额维护属于核心事务边界。
2. 确认周期账单入账时，应进入正式账单创建事务。
3. AI 解析记录保存和正式账单生成应分离，避免 AI 失败影响记账。
4. 通知生成、统计缓存失效和异步任务投递可在核心事务完成后处理。

## 5. WebSocket 预留位置

v1 可不启用 WebSocket，但包结构中可在 infrastructure 下预留 websocket 适配层。业务模块只产生通知事件，不直接依赖 WebSocket 实现。后续启用时由通知服务统一推送。
