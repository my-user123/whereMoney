# 01-系统整体架构

## 1. 架构目标

本系统第一阶段建设面向单人用户的 AI 智能记账与个人财务分析平台。整体架构需要支撑收入支出记账、分类、账户、预算、储蓄目标、周期账单提醒、统计可视化、AI 财务分析、月报导出和提醒能力。

架构设计目标：

1. 基础记账能力稳定可靠，AI 不可用时不影响核心账本功能。
2. 用户财务数据严格隔离，所有业务访问都以用户身份为边界。
3. 统计、导出、AI 分析等耗时能力与主交易链路解耦。
4. 移动端优先，首页仪表盘、记账入口和 AI 助手体验保持轻量。
5. 预留后续扩展能力，包括实时通知、账单导入、OCR、家庭账本和资产管理。

## 2. 总体分层

系统采用前后端分离架构，由前端应用、后端服务、数据库、缓存、消息队列、AI 外部服务、文件/导出存储和统一入口组成。

```text
用户浏览器 / 移动端 Web
        |
        v
Nginx 统一入口
  |             |
  v             v
前端应用       后端 API 服务
                  |
                  +-- PostgreSQL：核心业务数据
                  +-- Redis：缓存、临时状态、限流、任务状态
                  +-- RabbitMQ：异步任务、通知、导出、AI 分析
                  +-- AI 服务：自然语言解析、总结、建议、问答
                  +-- 文件存储：Excel/PDF 月报文件
```

## 3. 核心业务域

后端按业务域划分模块：

1. 用户与认证域：注册、登录、用户资料、默认货币。
2. 账本基础域：账户、分类、收入支出账单。
3. 预算与目标域：预算、储蓄目标、预算执行。
4. 提醒域：周期账单提醒、预算提醒、月度总结提醒。
5. 统计域：仪表盘、趋势、分类占比、热力图、报表数据。
6. AI 智能域：自然语言记账、月度总结、行为分析、预算建议、趋势预测、财务问答。
7. 导出域：Excel/PDF 月报导出。
8. 通知域：站内通知、任务完成提醒。

## 4. 核心链路

### 4.1 记账链路

用户通过手动表单或 AI 自然语言解析创建收入/支出账单。账单保存后，系统同步更新与账单强一致相关的数据，例如账户余额；统计报表可按查询实时计算或通过快照/缓存优化。

### 4.2 AI 分析链路

AI 分析由用户手动触发。后端汇总用户授权范围内的数据，向 AI 服务发起分析请求，保存 AI 报告或问答记录。AI 失败时需要返回可理解的失败状态，不影响账单、预算和统计等基础功能。

### 4.3 提醒链路

周期账单、预算阈值和月度总结提醒由系统生成站内通知。周期账单提醒只提醒用户确认，不自动生成正式账单。用户确认后才进入正式记账链路。

### 4.4 导出链路

Excel/PDF 月报导出可能耗时较长，适合作为异步任务处理。用户发起导出后，系统创建导出任务，后台生成文件，完成后通过通知或状态查询告知用户。

## 5. WebSocket 实时通信评估

### 5.1 v1 是否必须使用 WebSocket

v1 不强制引入 WebSocket。

原因：

1. 第一阶段是单人个人记账，没有多人协作、实时评论、在线状态等强实时场景。
2. 账单、预算、储蓄目标和仪表盘数据主要由用户主动操作触发，普通 REST 请求即可完成状态同步。
3. AI 分析和月报导出虽然存在后台任务进度，但 v1 可以通过任务状态查询、页面轮询和站内通知满足需求。
4. 周期账单提醒不要求秒级推送，用户进入系统后看到站内提醒即可满足第一阶段体验。

### 5.2 v1 应预留的实时扩展点

虽然 v1 可不实现 WebSocket，但后续很可能需要实时通知和后台任务进度，因此架构应预留 `/ws/` 入口和通知事件模型。

建议预留场景：

1. AI 报告生成完成后实时推送。
2. Excel/PDF 导出进度和完成状态实时推送。
3. 周期账单到期提醒实时推送。
4. 预算超额或接近超额提醒实时推送。
5. 多端登录时的账本状态同步。
6. 后续家庭共享账本中的多人实时变更通知。

### 5.3 实时通信架构原则

1. WebSocket 只作为通知和状态推送通道，不作为业务事实来源。
2. 所有核心数据变更仍以 REST API 和数据库事务为准。
3. WebSocket 事件需要能被站内通知兜底，避免用户离线时消息丢失。
4. Nginx 预留 `/ws/` 路由到后端 WebSocket 服务。

## 6. 可靠性与一致性原则

1. 账单创建、编辑、删除与账户余额更新应在服务边界内保持一致。
2. 统计结果和 AI 报告允许最终一致，但需要能识别数据失效。
3. 用户删除账单后，相关统计缓存需要失效，相关 AI 报告应删除或标记失效。
4. 异步任务失败时应保存失败状态，并允许用户重试。
5. AI 分析输出必须带风险提示，不能替代专业金融建议。

## 7. 部署视图

第一阶段本地开发和部署可采用以下组件：

1. Nginx：统一入口，转发前端、API 和预留 WebSocket。
2. 前端应用：提供移动端优先的 Web 体验。
3. 后端应用：提供 REST API、认证、业务逻辑、统计、异步任务入口。
4. PostgreSQL：保存核心业务数据。
5. Redis：保存缓存、临时任务状态、限流计数等。
6. RabbitMQ：处理通知、导出、AI 分析等异步任务。
7. 文件存储：保存生成的 Excel/PDF 月报。
