# 11-实时通信设计

## 1. 实时通信需求评估

根据第一阶段需求，系统包含提醒系统、AI 分析、Excel/PDF 月报导出和仪表盘刷新等场景。这些场景存在“状态变化通知”的需求，但不要求强实时协同。

第一阶段不包含以下强实时场景：

1. 多人协作账本。
2. 实时评论。
3. 在线状态展示。
4. 多人共同编辑。
5. 秒级交易同步。

## 2. v1 结论

v1 可以暂不实现 WebSocket。

v1 推荐方式：

1. 站内通知保存所有提醒和任务完成状态。
2. AI 分析和月报导出通过任务状态查询或短轮询展示进度。
3. 仪表盘在用户完成记账、删除账单、确认周期账单后主动刷新。
4. 周期账单提醒进入通知中心，不依赖实时推送。

## 3. 为什么不在 v1 强制实现

1. 单人记账场景对秒级推送依赖不高。
2. 引入 WebSocket 会增加认证、连接管理、断线重连、消息兜底和部署复杂度。
3. v1 的主要价值在记账、统计、预算和 AI 分析，实时推送不是核心阻塞点。
4. 站内通知加轮询足以覆盖导出和 AI 任务的初始体验。

## 4. 必须预留的扩展点

后续很可能需要 WebSocket，因此架构需要预留：

1. Nginx `/ws/` 路由。
2. 后端 websocket infrastructure 包。
3. 通知事件模型。
4. 任务状态事件模型。
5. 前端实时连接封装。
6. 站内通知作为离线兜底。

## 5. 后续启用场景

### 5.1 实时通知

预算超额、周期账单到期、月度总结提醒可实时推送到前端。

### 5.2 后台任务进度

AI 报告生成、问答响应、月报导出等后台任务可推送进度和完成状态。

### 5.3 多端同步

同一用户在多个设备上登录时，一个设备新增账单后，其他设备可收到刷新信号。

### 5.4 家庭账本扩展

如果后续支持家庭共享账本，WebSocket 可用于成员间账单变更提醒、共享预算变化和成员操作提示。

## 6. 实时通信原则

1. WebSocket 是推送通道，不是业务事实来源。
2. REST API 仍负责核心写操作。
3. 数据库和站内通知负责消息可追溯。
4. WebSocket 消息丢失时，用户仍可通过通知中心和状态查询获得结果。
5. WebSocket 连接必须基于用户身份认证。
