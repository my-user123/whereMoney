## 2. 视觉方向

### 2.1 设计定位

前端第一阶段采用“移动端优先的个人财务工作台”作为主视觉方向。首页不是营销页，而是用户每天打开后直接理解财务状态、快速记账、查看提醒和触发 AI 分析的主操作界面。

整体体验关键词：

1. 温暖克制。
2. 移动端优先。
3. 财务信息高密度但可扫描。
4. AI 洞察自然嵌入，不喧宾夺主。
5. 以仪表盘为中心串联记账、预算、储蓄、提醒和分析。

### 2.2 色彩与材质

遵守 `DESIGN.md`：

1. 页面背景使用暖奶油色 `#f7f4ed`。
2. 主文字使用炭黑 `#1c1c1c`。
3. 辅助文字使用 `#5f5f5d` 或炭黑透明度。
4. 卡片和分区主要依赖 `#eceae4` 边框，不使用重阴影。
5. 图表可使用少量克制的语义色：收入偏绿色，支出偏暖橙，预算预警偏琥珀，错误偏红，但不得形成高饱和大面积色块。

### 2.3 排版

1. 使用 `Camera Plain Variable` 或系统 sans-serif fallback。
2. 首页大数字可使用较大字号，但不做营销式巨型标题。
3. 工具栏、表单、列表和图表标签保持紧凑可读。
4. 不使用负 letter-spacing 作为普通 UI 文本样式。

### 2.4 组件风格

1. 卡片圆角控制在 8px 到 12px。
2. 主按钮为炭黑底、浅色文字、6px 圆角、轻微 inset shadow。
3. 次按钮为透明底、炭黑边框。
4. 图标按钮使用圆形或小圆角按钮，并配 tooltip。
5. 页面不使用卡片套卡片；同一视觉块内用分隔线、间距或列表行组织信息。

9. 月报导出
10. 设置

### 3.2 首页工作台结构

首页是 v1 的核心页面，移动端从上到下：

1. 顶部栏：产品名、当前月份/周期、通知入口、默认币种切换。
2. 财务摘要：收入、支出、结余、储蓄率。
3. 快速操作：记一笔、AI 记账、导出月报、生成总结。
4. 预算进度：总预算和重点分类预算。
5. 储蓄目标：目标进度和预计达成状态。
6. AI 洞察：最新有效报告摘要或生成入口。
7. 图表预览：趋势、小型分类占比、热力图入口。
8. 近期账单：最近 5 条。
9. 到期提醒：周期账单确认入口。

## 5. 页面设计

### 5.1 登录页 `/auth/login`

页面目标：用户登录系统。

主要内容：

1. 用户名输入。
2. 密码输入。
3. 登录按钮。
4. 注册入口。

API 依赖：

1. `POST /api/v1/auth/login`

状态：

1. loading：提交中，按钮显示加载态。
2. empty：输入为空时本地校验提示。
3. error：账号密码错误、网络异常。
4. success：登录成功后跳转 `/dashboard`。
5. submitting：防重复提交。

### 5.2 注册页 `/auth/register`

页面目标：新用户创建账号并完成基础初始化。

主要内容：

1. 用户名。
2. 密码。
3. 昵称。
4. 默认币种。
5. 注册按钮。

API 依赖：

1. `POST /api/v1/auth/register`

状态：

1. loading/submitting：注册中。
2. empty：必填项缺失。
3. error：用户名重复、密码不符合规则、网络异常。
4. success：注册成功后进入 `/dashboard`。

### 5.3 首页工作台 `/dashboard`

页面目标：让用户快速理解本期财务状态，并进入关键操作。

主要区域：

1. 顶部栏：月份/周期、币种、通知入口。
2. 财务摘要：收入、支出、结余、储蓄率。
3. 快捷操作：手动记账、AI 记账、生成月度总结、导出月报。
4. 预算进度：总预算、重点分类预算、超额提醒。
5. 储蓄目标：当前进度、剩余金额、预计压力。
6. AI 洞察卡片：最新总结摘要或生成入口。
7. 图表预览：趋势、分类占比、热力图。
8. 近期账单。
9. 周期账单到期提醒。

API 依赖：

1. `GET /api/v1/users/me`
2. `GET /api/v1/analytics/dashboard`
3. `GET /api/v1/analytics/trends`
4. `GET /api/v1/analytics/category-shares`
5. `GET /api/v1/notifications/unread-count`
6. `GET /api/v1/recurring-bill-instances`
7. `POST /api/v1/ai/reports`
8. `POST /api/v1/export-records`

状态：

1. loading：骨架屏，摘要、卡片、图表分别占位。
2. empty：无账单时展示轻量空状态，引导“记一笔”或“AI 记账”。
3. error：仪表盘加载失败，提供重试。
4. success：展示摘要、图表、提醒和近期账单。
5. submitting：生成 AI 报告、创建导出任务时按钮进入任务态。

实时/轮询：

1. v1 不依赖 WebSocket。
2. AI 报告和导出任务创建后，通过 REST 查询或通知中心反馈结果。
3. 后续如启用 `ai_report.updated`、`export_record.updated`、`notification.created`、`dashboard.refresh_required`，首页收到事件后重新拉取对应 REST 数据。
4. 断线或未启用 WebSocket 时回退为手动刷新和短轮询。

### 5.4 账单列表 `/transactions`

页面目标：查询、筛选、编辑和删除账单。

主要区域：

1. 顶部筛选：日期范围、类型、币种、账户、分类。
2. 收支汇总条。
3. 按日期分组的账单列表。
4. 移动端浮动记账按钮。
5. 桌面端可使用列表/表格布局。

API 依赖：

1. `GET /api/v1/transactions`
2. `GET /api/v1/accounts`
3. `GET /api/v1/categories`
4. `DELETE /api/v1/transactions/{transactionId}`

状态：

1. loading：列表骨架。
2. empty：筛选结果为空或暂无账单。
3. error：加载失败或删除失败。
4. success：展示分组列表。
5. submitting：删除确认中、筛选刷新中。

6. 发生时间。

API 依赖：

1. `POST /api/v1/transactions`
2. `GET /api/v1/transactions/{transactionId}`
3. `PUT /api/v1/transactions/{transactionId}`
4. `GET /api/v1/accounts`
5. `GET /api/v1/categories`

状态：

1. loading：编辑页加载详情。
2. empty：账户或分类为空时提示先创建。
3. error：保存失败、字段校验失败。
4. success：保存成功后返回账单列表或首页。
5. submitting：提交中禁用按钮。

### 5.6 AI 助手 `/ai`

页面目标：提供自然语言记账、AI 问答和分析入口。

主要区域：

1. 自然语言输入区。
2. AI 解析候选结果确认区。
3. 财务问答输入区。
4. 最近问答列表。
5. 月度总结、预算建议、趋势预测入口。

API 依赖：

1. `POST /api/v1/ai/transaction-parses`
2. `POST /api/v1/ai/transaction-parses/{parseRecordId}/confirm`
3. `POST /api/v1/ai/chat-records`
4. `GET /api/v1/ai/chat-records`
5. `DELETE /api/v1/ai/chat-records/{chatRecordId}`
6. `POST /api/v1/ai/reports`

状态：

1. loading：历史问答加载中。
2. empty：暂无问答或暂无解析结果。
3. error：AI 解析失败、问答失败、确认入账失败。
4. success：解析结果展示、问答结果展示、入账成功提示。
5. submitting：解析中、确认入账中、问答生成中。

实时/轮询：

1. v1 即时问答可以同步返回；较重报告使用任务模型。
2. 后续如启用 `ai_report.updated`，AI 页面收到事件后重新拉取报告详情或报告列表。

## 7. 响应式方案

### 7.1 移动端

1. 单列布局。
2. 底部导航固定。
3. 关键操作使用底部抽屉。
4. 首页摘要和图表使用纵向卡片。
5. 表格转换为列表行。
6. 筛选条件使用抽屉或横向滚动筛选 chips。

### 7.2 平板端

1. 首页可采用双列卡片布局。
2. 图表预览与预算/储蓄区域并排。
3. 筛选栏可固定在页面顶部。

### 7.3 桌面端

1. 左侧导航 + 主内容区域。
2. 首页使用 12 栅格布局。
3. 账单页可展示表格视图。
4. AI 助手可作为右侧面板或独立页面。
5. 统计页可使用多图表网格。

## 8. 状态设计总表

| 状态 | 适用场景 | 设计方式 |
| --- | --- | --- |
| loading | 页面初次加载、图表加载 | 骨架屏，保持布局稳定 |
| empty | 无账单、无预算、无报告、无通知 | 简短提示 + 主操作按钮 |
| error | API 失败、任务失败 | 错误文案 + 重试按钮 |
| success | 数据加载成功、保存成功 | 正常内容或轻量 toast |
| submitting | 表单提交、删除、确认入账 | 禁用按钮 + 加载指示 |
| processing | AI 报告、导出任务生成中 | 状态行 + 可刷新/自动轮询 |
| stale | AI 报告因账单变更失效 | 失效提示 + 重新生成入口 |

## 9. WebSocket 与降级设计

### 9.1 v1 设计

v1 不强制实现 WebSocket，使用 REST 查询和短轮询：

1. 通知数量：进入首页、操作完成后刷新。
2. AI 报告：任务创建后查询报告详情或列表状态。
3. 导出任务：任务创建后短轮询导出详情。
4. 仪表盘：账单新增、编辑、删除、周期账单确认后主动重新拉取。

### 9.2 后续实时事件

如后续启用 WebSocket，前端只接收轻量事件，然后重新拉取 REST 数据：

| WebSocket 事件 | 来源 | 前端动作 |
| --- | --- | --- |
| notification.created | 通知服务 | 刷新未读数量或通知列表 |
| ai_report.updated | AI 报告任务 | 重新拉取报告详情 |
| export_record.updated | 导出任务 | 重新拉取导出详情 |
| dashboard.refresh_required | 账单变更事件 | 重新拉取首页仪表盘 |
| recurring_bill.notified | 周期提醒任务 | 重新拉取到期提醒实例 |

### 9.3 断线降级

1. WebSocket 断线后保留页面可用性。
2. 自动切回 REST 查询或短轮询。
3. 用户可手动刷新。
4. 站内通知作为所有实时消息的离线兜底。

## 10. 前端数据与状态管理

### 10.1 服务端状态

适合使用查询缓存管理：

1. 用户资料。
2. 账户列表。
3. 分类列表。
4. 账单列表。
5. 预算列表。
6. 储蓄目标列表。
7. 仪表盘数据。
8. 图表数据。
9. 通知列表。
10. AI 报告和导出任务。

### 10.2 客户端状态

适合放在客户端状态：

1. 当前选中的周期。
2. 当前币种。
3. 移动端抽屉打开状态。
4. 当前筛选条件。
5. AI 输入草稿。
6. 临时表单状态。

### 10.3 缓存失效

以下操作后需要失效相关查询：

1. 新增、编辑、删除账单：失效账单、账户、仪表盘、统计、预算、储蓄目标、AI 报告列表。
2. 修改预算：失效预算和仪表盘。
3. 修改储蓄目标：失效储蓄目标和仪表盘。
4. 确认周期账单：失效周期实例、账单、账户、仪表盘、统计。
5. 创建 AI 报告：失效报告列表和通知。
6. 创建导出任务：失效导出列表和通知。

## 11. API 不足与待对齐问题

1. 首页 `GET /api/v1/analytics/dashboard` 的 `budgetHighlights`、`savingGoalHighlights`、`recentTransactions`、`aiInsightSummary` 需要进一步明确响应结构。
2. `GET /api/v1/analytics/trends`、`category-shares`、`heatmap`、`category-rankings` 需要明确图表数据点结构。
3. AI 报告 `content` 需要确定是 Markdown、纯文本还是结构化 JSON。前端建议使用 Markdown 或结构化块，以便展示段落、列表和风险提示。
4. 多币种统计默认按币种分组，前端需要 API 明确不传 `currency` 时的响应结构。
5. 导出任务是否返回进度百分比未确定；前端可先按状态展示。
6. 注册后默认账户和默认分类初始化策略需要明确，否则新用户首页空状态和记账表单需要额外引导。
7. 周期账单实例列表响应结构需要包含提醒规则名称、默认金额、币种、账户、分类和预计发生日期。
8. 预算提醒阈值已在 API 中设计为可配置，但默认值和可选范围需要后端确认。
